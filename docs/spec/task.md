# Техническое задание ра разработку CollegeLMS

## Стек разработки:  

- Postgres, .NET, Next.js, Nginx

## Архитектура: модульный монолит (понятие «сервис» → модуль)

## Развертывание

- При разработке будет использован docker-compose. 
- Развертывание будет на отдельном VPS (cloud.ru) через СI/CD Github Action


## Основные сервисы системы (это не микросервисы, это общее понятие):

Базовые сервисы (5 сервисов):

 - сервис сайта колледжа
 - сервис аутентификации и авторизации
 - сервис расписания занятий
 - сервис тестирования
 - сервис учебного процесса

Опциональные сервисы:

 - сервис электронного журнала 
 - сервис уведомлений
 - сервис генерации отчетов
 - сервис трасляции
 - сервис безопасности
 - сервис логгирования
 - сервис резервного копирования

1. Сервис сайта колледжа (SiteService)

Базовые требования:
- перенос данных с текущего сайта stvcc.ru
- новый сайт будет на Next.js.
- использовать компоненты Tailwind (версия 4)
- дизайн должен соответствовать обязательно цветам логотипа колледжа. Логотип в формате svg в import
- сайт должен быть адаптивен под все разрешения

Текущая структура сайта и данные для анализа:

http://stvcc.ru/sitemap.xml
http://stvcc.ru/sitemap-misc.xml 
http://stvcc.ru/category-sitemap.xml
http://stvcc.ru/page-sitemap.xml

Все типы контента и маршруты 
http://stvcc.ru/wp-json/wp/v2/

http://stvcc.ru/wp-json/wp/v2/categories?per_page=100
http://stvcc.ru/wp-json/wp/v2/posts?per_page=100
http://stvcc.ru/wp-json/wp/v2/pages?per_page=100
http://stvcc.ru/wp-json/wp/v2/tags?per_page=100
http://stvcc.ru/wp-json/wp/v2/users?per_page=100

Для  определения меню:
http://stvcc.ru/wp-json/wp/v2/pages?per_page=100&_fields=id,parent,title,slug,link


2. Сервис аутентификации и авторизации (AuthService)

Базовые требования:
- авторизация по логин (не почте) и паролю
- ролевая модель: админ, преподаватель, студент, диспетчер.


3. Сервис расписания занятий (ScheduleService)

Расписание формирует диспетчер вручную, данные получаются в виде файлов для каждой группы и преподавателя. Пример файлов прилагается (example_schedule_teacher.jpg, example_schedule_full.pdf).

Базовые требования:
- просмотр расписания по группе, преподавателю  
- редактирование (только диспетчер, админ).
- можно смотреть расписание на день, неделю, месяц, год, календарный вид
- экспорт расписания в pdf, excel

4. Сервис учебного процесса (LearningService)

В систему студентов, группы, курсы добавляет администратор или преподаватель

Базовые требования:

- управление студентами
- управление группами
- управление преподавателями
- создание учебных курсов (лекции, практики, тесты) — тесты через вызов TestingService
- личные кабинеты студентов и преподавателей
- загрузка файлов (PDF, ppt, pptx, doc, docx, mp4) — делегируется FileService

5. Сервис тестирования (TestingService)

Базовые требования:
- создание и проведение тестов
- LearningService вызывает TestingService для создания тестов в курсах


Служебные опциональные сервисы (с 6 по 19):

6. Сервис уведомлений  (NoticeService)
- каналы: email, push-уведомления (мобильное приложение), SMS, боты
- асинхронная отправка через RabbitMQ или Nuts


7. Сервис генерации отчетов (ReportService)
- формат: PDF, Excel. 
- построение графиков и диаграмм 


8. Сервис трансляции  (StreamService)
- преподаватель запускает стрим с лекцией
- студенты подключаются к лекции
- запись лекции можно сохранить

9. Сервис безопасности (SecurityService)
- защита от DDos атак
- внедрение капчи
- 2 факторная авторизация

10. Сервис логгирования  (LogService)
- все модули шлют логи в брокер сообщений → централизованный сбор и анализ
- на бекенде работает Serilog

11. Сервис резервного копирования (BackupService)
- бекап базы данных раз в неделю


12. Сервис хранения файлов (FileService)
- внешнее хранилище файлов на базе MinIO (self-hosted)
- LearningService вызывает FileService для загрузки/хранения файлов

13. Сервис электронного журнала (JournalService)

Базовые требования:
- ведение журнала на каждой паре
- учет посещаемости
- контроль даты проведения занятий
- контроль тем занятий

14.  Сервис классного руководителя (CuratorService)

Базовые требования:
- учет группы
- мероприятия
- характеристики
- отчеты об успеваемости

15. Сервис успеваемости (PerformService)

Базовые требования:
- управление ведомостями,
- контроль задолжностей
- управление зачетными книжками. 
- учёт выпускников

16. Сервис приемной комиссии (AdmissionService)
- контроль абитуриентов

17. Сервис отдела кадров (HrService)
- учет личных дел сотрудников

18. Telegram-бот (CollegeLMS.TelegramBot)

Базовые требования:
- бот Max (Макс): просмотр расписания по группе/преподавателю, inline-кнопки навигации
- бот Telegram: интерфейс для взаимодействия с OpenCode (установлен на VPS отдельно от compose)
- интеграция с OpenCode Server API (HTTP + SSE) — OpenCode как внешний dev-tool, не сервис LMS
- диспетчер может управлять расписанием через веб-приложение и рассылать уведомления группам через ботов
- запросы разрешений агента прокидываются в мессенджер (inline keyboard Allow/Deny)
- конфигурация через env vars, Docker контейнер в docker-compose

Стек: .NET 10, Telegram.Bot NuGet, Max REST API

19. Сервис аналитики сайта (AnalyticsService)

Техническое решение: комбинированный трекинг (nginx + JS)
- nginx access logs → базовые метрики (посещения, RPS, 4xx/5xx, гео)
- JS-сниппет → поисковые запросы, глубина просмотра, устройства
- POST /api/analytics/event — приём событий от клиента

Базовые требования:
- ежедневная статистика: посетители, просмотры страниц
- источники трафика (реферер): поисковик, соцсети, прямой
- устройства (User-Agent → desktop/mobile/tablet)
- время на сайте, глубина просмотра
- популярные страницы и разделы
- нагрузка на сервер: RPS, время ответа, 4xx/5xx
- поисковые запросы (сбор через JS, топ запросов, запросы без результатов)
- хранение: 6 месяцев
- анонимизация IP (после геолокации)
- дашборд для администратора
- экспорт отчёта за период (PDF/Excel)
- UI-аудит существующего поиска на сайте: удобство, автоподсказки, пустые состояния, результаты
